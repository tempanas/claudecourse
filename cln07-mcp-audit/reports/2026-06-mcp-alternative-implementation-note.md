# Примечание о реализации альтернативы MCP

## Почему Claude Code MCP не использовался

Claude Code MCP не использовался, потому что в рамках задания нет оплаченного доступа к Claude Code. Подключение и демонстрация MCP именно через эту среду поэтому недоступны.

Вместо имитации несуществующего подключения выбран воспроизводимый вариант на доступных инструментах: desktop Codex и Playwright.

## Как Codex и Playwright заменили браузерную часть задания

Desktop Codex выступил агентной средой, которая:

- прочитала структуру локального проекта;
- получила доступ к внешнему браузерному инструменту;
- запустила автоматизированную проверку;
- открыла локальный лендинг;
- проверила наличие ключевых элементов;
- сохранила скриншот;
- сформировала отчёт по результатам.

Playwright выступил браузерным инструментом. Он загружает страницу в Chromium и возвращает агенту проверяемый результат, а не только текст исходных файлов.

По смыслу это соответствует браузерной части MCP-демонстрации: языковая модель расширяет свои возможности подключённым инструментом и использует его для взаимодействия с внешней средой. Отличие заключается в том, что Playwright вызывается из среды Codex напрямую, а не через сервер MCP Claude Code.

## Практический результат

Созданы:

- автоматизированный скрипт `scripts/check_landing_with_playwright.js`;
- скриншот `temp/consultation-landing-screenshot.png`;
- отчёт `reports/2026-06-playwright-consultation-landing-audit.md`;
- данное пояснение об альтернативной реализации.

Скрипт проверяет загрузку локального сайта, наличие основных текстов и ошибки страницы. Он сначала пытается открыть `http://127.0.0.1:8080`, а если сервер недоступен, открывает локальный `index.html`.

Проект не публикуется в интернете и не передаёт данные внешним сервисам.

## Ограничения варианта

- это не демонстрация протокола MCP и не подключение MCP-сервера;
- результат показывает функциональный аналог браузерного инструмента, но не архитектуру Claude Code MCP;
- Playwright доступен в bundled-окружении desktop Codex и использует установленный системный Google Chrome; в обычном Node.js на другом компьютере пакет и совместимый браузер могут потребовать отдельной установки;
- проверка сфокусирована на одном локальном лендинге и не покрывает полноценное кроссбраузерное тестирование;
- автоматический аудит не заменяет ручную оценку дизайна, содержания и доступности;
- скриншот отражает страницу в Chromium с заданным размером viewport.

## Вывод

Codex + Playwright дают корректную и честную альтернативу для выполнения практической части задания: агент получает внешний браузерный инструмент, использует его для проверки локального сайта и создаёт проверяемые артефакты. При этом вариант не выдаётся за Claude Code MCP и явно фиксирует свои технические ограничения.
