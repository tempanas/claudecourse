# Consultation Request Triage

Разбирать только обезличенные заявки на консультацию. Не отправлять сообщения, не создавать реальные встречи, не проверять внешние системы и не выполнять другие внешние действия. Результат работы - анализ текста и локальный Markdown-отчёт.

## Входные данные

1. Прочитать заявку пользователя или указанный локальный файл с заявкой.
2. При необходимости свериться с `docs/consultation-reglament-summary.md`.
3. Не извлекать и не переносить в отчёт реальные ФИО, email, телефоны, ссылки на профили, Telegram, Zoom, таблицы или оплату.
4. Если во входе обнаружены персональные данные или реальные ссылки, заменить их нейтральными маркерами: `[REDACTED_NAME]`, `[REDACTED_EMAIL]`, `[REDACTED_PHONE]`, `[REDACTED_LINK]`.

## Обязательная проверка

Для стандартной заявки проверить наличие:

- курса или программы;
- урока;
- тарифа;
- потока;
- списка вопросов;
- предпочтительного времени;
- указания, нужен куратор или эксперт.

Считать список вопросов:

- `yes`, если вопросы сформулированы конкретно и позволяют подготовиться;
- `partial`, если указана только тема, часть вопросов или слишком общее описание;
- `no`, если вопросов нет.

Для тарифов VIP, VIP+, LUXURY и Корпоративный применять отдельную логику: если запрос пришёл от VIP-менеджера и требуется только Zoom, анализ профиля, курса, урока, потока и списка вопросов не выполнять и не считать эти поля критически недостающими. Проверить только, что запрос действительно относится к VIP-линейке и содержит достаточно демонстрационных сведений для внутренней фиксации задачи на создание Zoom. Не создавать реальную ссылку.

## Классификация тарифа

Нормализовать тариф в одну из групп:

1. `Базовый / Самостоятельный`.
2. `Бизнес / С поддержкой`.
3. `VIP / VIP+ / LUXURY / Корпоративный`.
4. `Не определён`, если тариф отсутствует или не совпадает с известными вариантами.

Не угадывать тариф по косвенным признакам.

## Предварительный тип консультации

Определять `consultation_type` консервативно:

- `paid` - для тарифа Базовый / Самостоятельный или если в заявке прямо указано, что консультация платная;
- `free` - для тарифа Бизнес / С поддержкой только при явном подтверждении, что консультация входит в курс, лимит не исчерпан и вопросы относятся к учебному процессу;
- `needs_review` - если для Бизнес / С поддержкой нет хотя бы одного подтверждения бесплатности, тариф не указан, данные противоречат друг другу или применяется VIP-логика, в которой Зерокодер19 не определяет оплату.

Для Бизнес / С поддержкой не объявлять консультацию бесплатной только на основании названия тарифа. Окончательное решение может требовать оценки куратора и проверки доступного лимита.

## Тип специалиста

Заполнять `specialist_type`:

- `curator`, если явно запрошен куратор;
- `expert`, если явно запрошен эксперт;
- `needs_review`, если специалист не указан, выбор не подтверждён или запрос относится только к созданию Zoom для VIP-менеджера.

## Риски и служебные проверки

Проверить риски:

- нет списка вопросов или он частичный;
- не указан тариф;
- не указан поток;
- для платной консультации нужна проверка оплаты;
- слот не указан или не согласован;
- Zoom ещё не создан;
- нет фиксации консультации в рабочей таблице.

Использовать следующие правила:

- `payment_check_required: yes` для предварительно платной консультации, если во входе нет безопасного текстового подтверждения оплаты;
- `payment_check_required: no` для бесплатной консультации, VIP-запроса только на Zoom или когда проверка оплаты не относится к текущему этапу;
- `zoom_required: yes`, если консультация должна состояться, но во входе нет явного подтверждения, что демонстрационная Zoom-ссылка уже подготовлена;
- `table_update_required: yes` для стандартных консультаций Базовый / Самостоятельный и Бизнес / С поддержкой, если фиксация не подтверждена;
- `table_update_required: no` для VIP-запроса только на Zoom, потому что фиксацию выполняет VIP-менеджер.

Определять `risk_level`:

- `low` - обязательные данные заполнены, тип консультации и специалист определены, нет блокирующих проверок;
- `medium` - есть один или несколько устранимых пробелов, но следующий шаг понятен;
- `high` - отсутствуют тариф или вопросы, есть противоречия, нельзя определить ответственного либо платная консультация может быть организована без проверки оплаты.

## Следующий шаг

Сформировать короткий упорядоченный список `next_actions_for_zerocoder19`. Включать только внутренние действия, например:

- уточнить отсутствующие данные;
- передать вопросы куратору на внутреннюю оценку;
- проверить лимит бесплатных консультаций;
- зафиксировать необходимость проверки оплаты;
- согласовать слот;
- подготовить демонстрационный плейсхолдер Zoom;
- отметить необходимость фиксации в таблице.

Не писать сообщения от имени Зерокодер19 и не обращаться к студенту, куратору, эксперту или VIP-менеджеру. Не выполнять перечисленные действия во внешних системах.

## Формат результата

Вернуть валидный JSON без дополнительных ключей по следующей схеме:

```json
{
  "request_summary": {
    "student_alias": "обезличенный идентификатор студента",
    "tariff": "тариф",
    "program": "курс или программа",
    "lesson": "урок",
    "questions_provided": "yes/no/partial",
    "preferred_slot": "указан или не указан"
  },
  "triage_result": {
    "consultation_type": "free/paid/needs_review",
    "specialist_type": "curator/expert/needs_review",
    "risk_level": "low/medium/high",
    "missing_data": ["список недостающих данных"],
    "payment_check_required": "yes/no",
    "zoom_required": "yes/no",
    "table_update_required": "yes/no"
  },
  "next_actions_for_zerocoder19": [
    "действие 1",
    "действие 2",
    "действие 3"
  ],
  "internal_comment": "краткий комментарий для внутренней работы"
}
```

Если значение неизвестно, использовать понятное безопасное значение `"не указано"` в текстовом поле и добавить название поля в `missing_data`. Для VIP-полей, которые не требуются по отдельной логике, использовать `"не требуется для VIP-запроса"` и не добавлять их в `missing_data`.

## Сохранение отчёта

1. Создать или обновить локальный файл `reports/2026-06-consultation-request-triage-result.md`.
2. Добавить заголовок `# Consultation Request Triage Result`.
3. Указать источник как обезличенную заявку без реальных ссылок и персональных данных.
4. Поместить итоговый JSON в блок `json`.
5. После JSON добавить раздел `## Краткий вывод` с одним абзацем о типе консультации, рисках и ближайшем внутреннем действии.
6. Не создавать другие отчёты, если пользователь явно не попросил об этом.

Перед завершением проверить, что отчёт находится внутри `reports/`, JSON синтаксически корректен, а в файле нет реальных персональных данных, контактов, ссылок или подтверждений внешних действий.
