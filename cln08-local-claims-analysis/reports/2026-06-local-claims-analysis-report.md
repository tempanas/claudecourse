# Отчёт: локальный анализ претензий студентов

## 1. Цель проекта

Цель проекта — автоматизировать первичную классификацию обращений студентов с помощью локальной языковой модели. Решение читает обезличенные обращения из CSV, определяет характер и срочность каждого обращения и сохраняет результат в структурированном виде для дальнейшей проверки сотрудником.

Автоматизация не принимает окончательных решений по претензиям и возвратам. Она помогает быстрее сортировать поток обращений и выбирать следующий рабочий шаг.

## 2. Почему выбрана локальная модель

Обращения студентов могут содержать чувствительный рабочий контекст. Локальная модель подходит для этой задачи, потому что:

- текст обрабатывается на компьютере пользователя;
- запросы отправляются только в локальный API Ollama `http://localhost:11434/api/generate`;
- не используются облачные API и внешние сервисы;
- можно контролировать модель, промпт и формат результата;
- исходные и итоговые данные остаются в локальных файлах проекта.

В учебном наборе используются только искусственные сообщения и псевдонимы вида `Студент-01`. Реальные ФИО, телефоны, email, ссылки и платёжные данные отсутствуют.

## 3. Использованная модель

Для анализа подготовлена и использована локальная модель `claims-llama-32k:latest`. Она создана на основе `llama3.2:3b`.

Если пользовательская модель не запускается, резервным вариантом является непосредственный запуск базовой модели:

```bash
python3 scripts/analyze_student_claims.py --model llama3.2:3b
```

## 4. Изменение параметров через Modelfile

Конфигурация пользовательской модели хранится в `Modelfile.claims-32k`. В ней увеличен размер контекстного окна:

```text
PARAMETER num_ctx 32768
```

Значение `num_ctx 32768` позволяет модели учитывать более длинный промпт и содержимое обращения в одном контексте. Для текущего набора такой объём избыточен, но он демонстрирует настройку локальной модели через Modelfile и оставляет запас для более длинных обращений.

## 5. Анализируемые данные

Исходный файл `data/2026-06-student-claims-demo-data.csv` содержит 10 обезличенных обращений за июнь 2026 года. В нём представлены жалобы на качество объяснения, запрос на возврат, негативная обратная связь, вопросы о консультациях и тарифах, техническая проблема, просьба связаться и VIP-сценарий.

Поля исходного набора:

- `claim_id` — идентификатор обращения;
- `date` — дата;
- `student_alias` — обезличенный псевдоним;
- `tariff` — тариф;
- `program` — программа;
- `message` — текст обращения;
- `has_refund_request` — наличие запроса на возврат;
- `has_contact_permission` — разрешение на контакт.

Файл `data/2026-06-student-claims-demo-data-analyzed.csv` содержит результат тестового запуска с ограничением `--limit 5`, поэтому в итоговой таблице присутствуют первые 5 из 10 исходных обращений.

## 6. Определяемые категории

Модель добавляет четыре поля:

- `sentiment` — тональность: `positive`, `neutral` или `negative`;
- `claim_type` — жалоба, возврат, консультация, техническая проблема, вопрос по курсу или смешанное обращение;
- `priority` — приоритет: `low`, `medium` или `high`;
- `action` — рекомендуемое следующее действие.

Допустимые действия: ответить, передать в претензионный сектор, проверить возврат, назначить консультацию, уточнить данные или проверить техническую проблему.

Все пять сохранённых результатов совпадают с fallback-набором `neutral,вопрос по курсу,medium,уточнить данные`. Это означает, что во время данного тестового запуска ответы модели не прошли строгую проверку допустимых значений. После запуска скрипт был усилен: английские варианты категорий нормализуются, а неизвестный ответ больше не останавливает обработку остальных строк.

## 8. Обращения с высоким приоритетом

В сохранённом analyzed CSV обращений с приоритетом `high` нет: все пять строк получили fallback-приоритет `medium`.

При проверке исходных признаков обращение `CLM-2026-002` должно иметь высокий приоритет, потому что содержит явный запрос на возврат денег. Обращение `CLM-2026-003` также требует внимательной проверки из-за выраженного негатива, хотя окончательный уровень приоритета должен подтвердить сотрудник.

## 9. Где нужен возврат

В исходном CSV запрос на возврат явно отмечен у `CLM-2026-002`: поле `has_refund_request` равно `true`, а в сообщении содержится просьба вернуть оплату.

Ожидаемая классификация по правилам проекта:

```text
negative,возврат,high,проверить возврат
```

В текущем analyzed CSV эта строка получила fallback, поэтому результат модели необходимо повторно проверить после обновления парсера.

## 10. Передача в претензионный сектор

В сохранённом analyzed CSV действие `передать в претензионный сектор` не назначено ни одному обращению.

По содержанию исходных данных кандидатом на передачу является `CLM-2026-003`: студент выражает сильное разочарование качеством обратной связи. Решение об эскалации должен подтвердить сотрудник претензионного сектора.

## 11. Назначение консультации

В сохранённом analyzed CSV действие `назначить консультацию` отсутствует.

В исходном наборе консультация явно требуется для:

- `CLM-2026-004` — вопрос о доступности консультации и записи на неё;
- `CLM-2026-010` — срочный VIP-сценарий с пропущенной персональной консультацией.

Строка `CLM-2026-010` не вошла в тестовый analyzed CSV из-за ограничения `--limit 5`.

## 12. Вывод для работы Зерокодер19

Зерокодер19 — название должности, а не имя человека. Локальная автоматизация помогает этой роли:

- быстрее находить возвраты и срочные обращения;
- отделять технические проблемы от вопросов по курсу;
- выявлять обращения для консультации или претензионного сектора;
- получать единый структурированный результат;
- сохранять обезличенные рабочие данные локально.

Тест показал не только пользу автоматизации, но и необходимость контролировать формат ответа небольшой локальной модели. Нормализация значений, fallback-логика и ручная проверка критичных случаев делают процесс устойчивее. Для итоговой оценки рекомендуется повторно обработать все 10 строк обновлённым скриптом и отдельно проверить возвраты, высокий приоритет и VIP-сценарии.
